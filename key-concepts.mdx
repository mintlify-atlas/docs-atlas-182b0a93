## Workflows

A workflow is a sequence of steps that runs automatically when an event occurs. Workflows are linear â€” each step runs in order, and every step can read the outputs of the steps that came before it.

```
Trigger â†’ Step 1 â†’ Step 2 â†’ Step 3 â†’ ...
```

Each workflow has exactly one **trigger** (the first step) and one or more **action or code steps** that follow.

[Read more about workflows](/platform/workflows)

## Triggers

Triggers start a workflow. Pipedream supports several trigger types:

<CardGroup cols={2}>
  <Card title="HTTP / Webhook" icon="globe">
    Pipedream generates a unique URL. Any HTTP request to that URL triggers the workflow immediately.
  </Card>
  <Card title="Timer" icon="clock">
    Run a workflow on a schedule â€” every 5 minutes, once a day, or with a custom cron expression.
  </Card>
  <Card title="App events" icon="bolt">
    Watch for new data in a connected app. For example, trigger when a new GitHub issue is opened or a new row appears in Google Sheets.
  </Card>
  <Card title="Email" icon="envelope">
    Pipedream generates a unique email address. Incoming emails trigger the workflow with the parsed message as the event.
  </Card>
</CardGroup>

The trigger event is always available in subsequent steps as `steps.trigger.event`.

## Steps

Steps are the units of work inside a workflow. There are two kinds:

- **Action steps** â€” pre-built components that perform a specific task (send a Slack message, add a row to Google Sheets, create a GitHub issue).
- **Code steps** â€” arbitrary code you write in Node.js, Python, Go, or Bash.

Steps run sequentially. A step has access to all exports from previous steps via the `steps` object.

## Props

Props are inputs you define on a component. They tell Pipedream what data a step needs at configuration time (when you set up the workflow) or at runtime.

```javascript
export default {
  name: "Action Demo",
  description: "This is a demo action",
  key: "action_demo",
  version: "0.0.1",
  type: "action",
  props: {
    name: {
      type: "string",
      label: "Name",
    },
  },
  async run() {
    return `hello ${this.name}!`;
  },
};
```

Common prop types:

| Type | Description |
|------|-------------|
| `string` | A text value |
| `integer` | A whole number |
| `boolean` | `true` or `false` |
| `string[]` | An array of strings |
| `app` | A connected app account (handles OAuth and API keys) |
| `$.interface.http` | Exposes an HTTP endpoint as a trigger |
| `$.interface.timer` | Runs the component on a schedule |

### App props

When a prop has `type: "app"`, Pipedream manages authentication for you â€” including OAuth token refresh. You access credentials through `this.appName.$auth`:

```javascript
import { WebClient } from "@slack/web-api";

export default defineComponent({
  props: {
    slack: {
      type: "app",
      app: "slack",
    },
  },
  async run({ steps, $ }) {
    const web = new WebClient(this.slack.$auth.oauth_access_token);
    return await web.chat.postMessage({
      text: "Hello, world!",
      channel: "#general",
    });
  },
});
```

## Exports and step references

Every step can export data that subsequent steps can read. There are two ways to export:

### Return value

Returning a value from `run()` exports it as the step's `$return_value`:

```javascript
export default defineComponent({
  async run({ steps, $ }) {
    return { userId: 42, name: "Alice" };
  },
});
```

Access it in the next step with `steps.stepName.$return_value`.

### Named exports

Use `$.export(key, value)` to export named values:

```javascript
export default defineComponent({
  async run({ steps, $ }) {
    $.export("userId", 42);
    $.export("name", "Alice");
  },
});
```

Access them with `steps.stepName.userId` and `steps.stepName.name`.

### Accessing trigger data

The trigger event is always available at `steps.trigger.event`. For an HTTP trigger, this includes `body`, `headers`, `method`, `path`, and `query`:

```javascript
export default defineComponent({
  async run({ steps, $ }) {
    const { body, headers } = steps.trigger.event;
    console.log(body.userId); // access request body fields
  },
});
```

## Event sources

Event sources are reusable components that watch for new data in an external service and emit events to trigger workflows. A source handles the polling, deduplication, and delivery logic so your workflow only sees new events.

[Read more about event sources](/platform/event-sources)

## Actions

Actions are pre-built, reusable code steps contributed by Pipedream and the community. You add an action to a workflow and configure its props â€” no boilerplate required.

[Read more about actions](/platform/actions)

## Destinations

Destinations are asynchronous delivery targets. Unlike regular step exports (which are synchronous), destinations send data in the background â€” useful for high-volume event forwarding to S3, Snowflake, or an HTTP endpoint.

```javascript
$send.s3({
  bucket: "your-bucket-here",
  prefix: "your-prefix/",
  payload: event.body,
});
```

[Read more about destinations](/platform/destinations)

## Ending a workflow early

Call `$.flow.exit()` (or the platform `$end()` function) to stop execution at any step without treating it as an error:

```javascript
export default defineComponent({
  async run({ steps, $ }) {
    if (!steps.trigger.event.body.userId) {
      // Stop the workflow â€” no further steps will run
      $.flow.exit("No userId provided");
    }
    // ... rest of logic
  },
});
```

## Connected accounts

Pipedream manages OAuth flows, token storage, and token refresh for 2,800+ apps. When you add an `app` prop to a component, Pipedream prompts you to connect an account in the UI. Credentials are then injected at runtime via `this.appName.$auth`.

<Note>
Credentials are never exposed in workflow logs or event data. Pipedream encrypts all stored credentials.
</Note>
